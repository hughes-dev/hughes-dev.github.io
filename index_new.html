System overview

The Prison System is a distributed robotics platform built to support robot patrol workflows. It combines robot-side autonomy and business logic with a server-side control plane (UI, database, streaming coordination, and local AI analysis).

  • Robot side handles navigation events, maps low-level nav signals to high-level locations, runs headcount/business logic, and streams media.
  • Server side hosts the operator UI and API, stores events in PostgreSQL, runs MediaMTX, and performs local LLM-based image analysis via Ollama.
  • Designed to evolve rapidly with demo-driven requirements while staying debuggable and deployable on constrained machines.

Design goals

  • Low-latency visibility: streaming + event timeline for operators.
  • Robustness: tolerate flaky networks and real-world “this will break” conditions.
  • Clear separation of concerns: robot does real-time business logic; server does coordination, persistence, UI, and analysis.
  • Auditability: store events + artifacts so you can answer “what happened?” after the fact.

Translation: fewer “magic scripts,” more explicit services with logs, configs, and predictable startup behavior.

Subsystems

One “project,” many services. I present it as a single flagship system, with drill-down into the parts that matter.

Robot: Location & Headcount Core Location mapping + headcount/business rules run on the robot now.

The robot receives navigation signals (arrivals at nodes) and translates them into a high-level location context stored/associated with patrol state. Headcounting logic and mode handling run on-device so decisions aren’t coupled to network conditions or server round trips.

The server serves the UI and API layer. It aggregates incoming events, provides operator visibility, and centralizes persistent storage. The goal is boring reliability: clear endpoints, consistent schemas, and logs you can actually debug at 2 AM.

MediaMTX provides the streaming backbone. The robot produces streams; the server exposes endpoints for consumption in the operator UI. The design focuses on predictable startup, clean config, and minimal moving parts.

System evolution

This system changed over time. That’s not a flaw — it’s the job. The key architectural shift was moving headcount + location handling from server-side logic to robot-side logic to improve reliability and reduce coupling.

Earlier iteration
Server did headcount + location logic
Faster to prototype, but server round-trips and coupling made “real-time” behavior fragile. Debugging also got messy because the server was pretending to be the robot’s brain.
Current architecture
Robot owns real-time business logic
Robot handles location mapping and headcount state. Server becomes the control plane: UI + DB + streaming coordination + local AI analysis. Cleaner boundaries, fewer failure cascades.
Ongoing
More analysis & artifacts, better operator UX
Refining the event timeline, adding analysis hooks, and improving repeatable deployment and operations (service management, logs, configs, and sane defaults).

Impact

  • Enabled live demos and pilot-style workflows by providing stable UI + streaming + event capture.
  • Improved reliability by moving real-time logic to the robot and reducing server coupling.
  • Created an auditable record of patrol runs with persisted events and analysis artifacts.
  • Made the system easier to operate with explicit services instead of “run this one script and pray.”

Constraints & tradeoffs

  • Can’t share code/demos: solved by writing clear system-level explanations and emphasizing design decisions.
  • Constrained hardware: local inference with predictable sizing (image downscaling, bounded latency).
  • Network variability: robot keeps core logic; server provides persistence and visibility.
  • Small team, fast iteration: architecture optimized for clarity and maintainability over buzzword compliance.

FAQ (the questions interviewers actually ask)

Why move headcount/location logic onto the robot? Reliability, latency, and clean system boundaries.

If the server owns real-time logic, network hiccups become business-logic failures. Moving location/headcount onto the robot reduced coupling, improved determinism, and made it easier to reason about state. The server then becomes what it should be: UI + persistence + analysis.

How do you handle “multi-service chaos” during deployment? Explicit services, predictable startup, and logs that tell the truth.

Multiple services per machine is manageable when each has a clear responsibility, config, and logging. The goal is minimizing hidden coupling: services communicate through defined APIs and persistent state, not shared assumptions.

What’s the LLM doing here? Local image analysis with structured results (not “AI fairy dust”).

The server uses local LLM inference (via Ollama) to analyze captured images and produce structured outputs that can be stored, reviewed, and tied back to patrol events. The design prioritizes bounded latency and predictable resource usage.

Why not just make everything cloud-native? Constraints: hardware, environment, and speed of iteration.

Not every project needs a cloud platform to be “real.” This system’s realities were local machines, constrained resources, and fast demo cycles. The focus was reliable services, clear interfaces, and operational simplicity.
